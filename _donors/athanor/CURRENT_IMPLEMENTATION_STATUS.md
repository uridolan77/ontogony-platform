## Permanent invariants

- PostgreSQL is canonical write authority
- Neo4j is projection/read model only
- Versions/events/snapshots are immutable/append-only
- `EvidenceSpan` is citation-grade provenance; `SourceChunk` is retrieval
- Embeddings/semantic search support discovery, not truth
- Tools/workflows/review inbox are operational surfaces, not canonical authority
- Agents/system actors may propose/recommend/detect/queue/execute non-canonizing operations
- Agent/system actors do not bypass project-local authority (no implicit canonization power from `actor_type`).
- Agents/system actors may not directly canonize (canonical-risk commands require a human actor).

- `actor.default_role` is only a hint
- `actor.actor_type = system` does not bypass project-local authority

## Authority rules (implemented)

Project-local authority comes from **active** `project_actor_role` rows (strongest-role precedence).

For **canonical-risk finalization operations**, role is **necessary but not sufficient**: the subject actor must also be **human** (`actor_type=user`; API alias `"human"`). `agent`, `system`, and `external_tool` actors may not directly perform canonical-risk finalization even if granted `owner`/`editor` in `project_actor_role`.

| Operation | Required effective project role |
|---|---|
| Create candidate object/version, bind evidence | `owner` \| `editor` \| `reviewer` \| `proposer` \| `extractor` |
| Accept object version | `owner` \| `editor` |
| Reject/defer object version | `owner` \| `editor` \| `reviewer` |
| Manage contradiction cases (create/add participant/resolve) | `owner` \| `editor` \| `reviewer` |
| Create candidate relation | `owner` \| `editor` \| `reviewer` \| `proposer` \| `extractor` |
| Accept relation | `owner` \| `editor` |
| Reject relation | `owner` \| `editor` \| `reviewer` |
| Create canonical snapshot | `owner` \| `editor` |
| List project epistemic ledger / ledger entry detail / metabolism read endpoints | any active project role (`viewer` included) |
| Append epistemic ledger row (`POST .../epistemic-ledger`) | `owner` \| `editor` |
| Upsert source assessment / trust (`PUT .../assessments`) | `owner` \| `editor` \| `reviewer` |
| Read source assessments (`GET .../assessments`, `GET .../effective`) | any active project role (`viewer` included) |
| Create, enable, disable, or evaluate maintenance policies | `owner` \| `editor` |
| GET revision transaction | any active project role (`viewer` included) |
| Read project health dashboard (`GET .../dashboard/health`) | any active project role (`viewer` included) |
| Open revision or attach surprise/object/version/review event/resolution event | `owner` \| `editor` \| `reviewer` \| `proposer` \| `extractor` |
| Propose or reject revision (container) | `owner` \| `editor` \| `reviewer` |
| Accept revision (container) | `owner` \| `editor` |
| Close revision | `owner` \| `editor` when closing an **accepted** revision; `owner` \| `editor` \| `reviewer` when closing from other active statuses (cancel path) |

Notes:

- `actor.default_role` is a **fallback/capability hint** only; it does not grant project-local canonization authority by itself.
- `actor.actor_type = system` does **not** automatically grant project-local authority.

## Snapshot hash compatibility note

Snapshot hash verification depends on the canonical hash payload used at creation time.

- Snapshots created **before** hash payload rules were stabilized may fail `verify-hash`.
- If you hit this, the intended fix is: **recreate the snapshot** (or backfill/recompute hashes in a controlled migration later).

PR16 policy note:

- `canonical_snapshot.snapshot_hash` is defined over the frozen snapshot payload (header + items + contradictions).
- `canonical_snapshot.synthesis_*` fields are enrichment and **do not** affect `verify-hash`.
