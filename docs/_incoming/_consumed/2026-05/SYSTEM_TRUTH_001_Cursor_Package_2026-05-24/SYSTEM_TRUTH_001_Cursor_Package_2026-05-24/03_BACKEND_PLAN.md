# 03 — Backend Implementation Plan

## A. Shared contracts

Prefer implementing shared DTOs in `ontogony-platform` if that is the existing home for cross-service hosting mechanics. If that would create too much churn, implement identical local DTOs first and add a platform extraction follow-up.

Contracts are provided in:

```text
contracts/health.v1.schema.json
contracts/ready.v1.schema.json
contracts/system-compatibility-manifest.schema.json
contracts/system-compatibility-summary.schema.json
```

## B. Health v1

Every service should return a JSON response from `/health` and `/health/live` with this minimum shape:

```json
{
  "schemaVersion": "health.v1",
  "status": "healthy",
  "service": "conexus",
  "serviceDisplayName": "Conexus",
  "version": "0.1.0-alpha.local",
  "baseline": "SYSTEM-ALPHA-006",
  "gitSha": "local-or-sha",
  "buildTimeUtc": "2026-05-24T00:00:00Z",
  "environment": "Development",
  "checkedAtUtc": "2026-05-24T00:00:00Z"
}
```

Status values:

```text
healthy | degraded | unhealthy | unknown
```

`/health` should not require privileged credentials.

## C. Ready v1

Every service should return detailed readiness from `/ready`:

```json
{
  "schemaVersion": "ready.v1",
  "status": "not_ready",
  "service": "conexus",
  "checkedAtUtc": "2026-05-24T00:00:00Z",
  "checks": [
    {
      "id": "conexus.provider.fake",
      "label": "Fake provider",
      "status": "ready",
      "severity": "required",
      "detail": "Runtime registered and local fake route available."
    },
    {
      "id": "conexus.provider.openai.credentials",
      "label": "OpenAI credentials",
      "status": "warning",
      "severity": "optional",
      "detail": "OpenAI provider registered but CONEXUS_PROVIDER_OPENAI_API_KEY is not configured."
    }
  ]
}
```

Status values:

```text
ready | not_ready | degraded | unknown
```

Check status values:

```text
ready | warning | failed | skipped | unknown | not_applicable
```

Severity values:

```text
required | optional | informational
```

## D. Conexus readiness detail

Conexus currently appears live/healthy but strict-not-ready. Make the reason visible.

Recommended Conexus checks:

```text
conexus.postgres.connection
conexus.postgres.migrations
conexus.projects.default
conexus.project_key.dev
conexus.provider_registry.fake
conexus.provider_registry.openai
conexus.provider_credentials.openai
conexus.routing.aliases
conexus.routing.project_overrides
conexus.route_preview.default
conexus.model_call_store
conexus.route_decision_store
conexus.telemetry.store
conexus.admin_auth
```

Important policy:

- Missing OpenAI credentials should **not** make local fake-provider readiness fail if OpenAI is optional.
- An enabled provider with missing credentials may make provider-readiness degraded/warning.
- An alias pointing to a missing or unregistered provider should make readiness fail.
- If Conexus can serve the configured local alias via fake, local operator usability should be `usable_with_warnings`, not `blocked`.

## G. Compatibility summary artifact

Produce a local generated summary, e.g.:

```text
docs/generated/operator-system-compatibility.json
```

or a frontend-accessible equivalent.

It should include:

- generated timestamp,
- baseline,
- expected service contracts,
- actual health/ready status,
- version metadata,
- data source classification,
- compatibility verdict,
- warnings,
- failures,
- staleness.

This artifact can be generated by script initially. Later it can be generated by CI/local validation.
