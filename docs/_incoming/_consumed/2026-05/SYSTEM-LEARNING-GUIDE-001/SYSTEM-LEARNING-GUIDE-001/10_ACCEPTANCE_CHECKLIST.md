# 10 — Acceptance checklist

- [ ] New developer can identify repo ownership boundaries.
- [ ] New developer can start the local system from docs alone.
- [ ] Developer can run governed fake E2E and inspect proof from docs alone.
- [ ] Developer can explain domain vs model purpose vs Conexus alias vs provider route.
- [ ] Developer can add/change an API route and know every artifact/check to update.
- [ ] Developer can find canonical guide instead of searching stale docs.
- [ ] Stale/duplicate docs identified and marked for cleanup.
- [ ] Generated docs linked, not manually duplicated.
- [ ] Commands correspond to actual scripts or are explicitly proposed.
- [ ] Guides do not claim production readiness beyond verified current state.
- [ ] Package includes exact file targets and test/check commands.
